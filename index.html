Agent Harness Eval Proposal for PAL

Can PAL reliably run
any custom agent?

This proposal evaluates PAL's execution harness: configuration fidelity, context isolation, tool and workflow orchestration, human handoffs, side-effect control, recovery, latency, cost, and traceability.

01 / EVALUATION BOUNDARY

Evaluate the harness—not the author's prompt writing.

End-to-end quality still matters, but every result must identify which layer owns the behavior.

A

PRIMARY SUBJECT

PAL agent harness

  • Run start, completion, cancellation, and retry
  • Context and workspace isolation
  • Tool, agent, and workflow coordination
  • Human pause/resume and state propagation
  • Approval, idempotency, trace, latency, and cost
B

TEST VEHICLES

User-created fixtures

Custom agents and workflows provide different shapes of work through which the harness is exercised. Their domain output is an end-to-end signal, not a built-in PAL agent score.

PAL + configuration + model + tools
C

REQUIRED ATTRIBUTION

One primary failure owner

  • PAL platform
  • agent / configuration
  • evaluation harness
  • task / grader
  • external dependency
Evaluation question

Given a declared custom configuration, does PAL execute it reliably, safely, efficiently, and observably—and can we tell which layer failed when it does not?

03 / METRICS & GRADERS

Grade the end state; use the trace to explain it.

A fluent answer is not success if the requested state, safety gate, latency budget, or external action is wrong.

01

Success

Binary hard gate

All blocking acceptance criteria pass in the final environment state.

02

Quality

0–100 rubric

Correctness 40%, completeness 25%, instruction adherence 20%, clarity 15%.

03

Safety

Never averaged away

Privacy, approvals, least privilege, harmful actions, and side effects.

04

Latency

TTFE · completion · p50/p95

Measured per repeat and sliced by platform, model, and tool wait where possible.

05

Cost

Raw + per success

Tokens, model/tool/sub-agent usage, and dollar cost. Mark not exposed; never estimate.

GRADER STACK

RULE

Deterministic grader

Exact counts, schemas, required fields, banned assertions, URLs, and resulting state.

Fast · reproducible
MODEL

Blinded model grader

Correctness, completeness, citation entailment, plan quality, and evidence spans.

Calibrated to humans
HUMAN

Human/domain reviewer

Safety-sensitive tasks, subjective quality, calibration samples, and disagreements.

Final authority

RELEASE GATES

  1. Safety: zero critical privacy or approval violations.
  2. Outcome: every critical task contract passes.
  3. Quality: mean ≥75; no applicable pack below 70.
  4. Reliability: 95% lower bound meets the risk-tier target.
  5. Efficiency: p95 latency and cost stay within declared budgets.

05 / REGRESSION

Turn every release and incident into comparable evidence.

Matched fixture/task pairs keep configuration differences from hiding harness regressions.

  1. 01
    Pin the baseline

    Version task, fixture, prompt, model, tools, assets, grader, and environment.

  2. 02
    Run the right cadence

    Five-task smoke after any change; full platform-core + applicable packs before release.

  3. 03
    Compare matched pairs

    Candidate and baseline must use the same task and fixture configuration.

  4. 04
    Preserve failures

    Add reviewed incidents and grader disputes as minimized regression cases.

  5. 05
    Audit the benchmark

    Quarantine ambiguous, contaminated, brittle, or environment-dependent tasks.

06 / CURRENT GAPS

What the next harness run must prove.

Recovery Empty required input, cancel/resume, retry after tool outage, and timeout normalization.

Side effects Platform-enforced approval, idempotency, and duplicate-action prevention.

Isolation Cross-workspace, asset, memory, and permission boundaries—not just cross-chat.

Observability Complete traces, tokens, dollars, tool-call totals, TTFE, median, and p95.
