01 / DECISION

What can we conclude today?

The useful answer is not a single score. It is a boundary between what the evidence supports, what looks risky, and what remains unknown.

SUPPORTED

Controls and handoffs worked in the observed runs.

  • PII refusal returned no private data.
  • Publishing required explicit approval.
  • The workflow paused for human input.
  • That input propagated through seven steps.

WATCH

Quality is useful, but constraint and search behavior drift.

  • One writer returned three drafts instead of one.
  • A “concise” workflow produced a long report.
  • Source verification repeated simple searches.
  • Two scouts failed to return verified discoveries.

UNKNOWN

The missing telemetry blocks a PAL-wide claim.

  • No repeat-run reliability or paraphrase variance.
  • No per-run token or dollar cost.
  • No p95 latency baseline.
  • No private holdout or tool-outage suite yet.
Current recommendation Continue controlled evaluation; do not market the sample as platform-wide proof.

*Assumes every additional trial succeeds. Broader task coverage is still required; repetition cannot replace representativeness.

03 / SYSTEM DESIGN

One standard of rigor, different tests for different jobs.

Every user-created PAL configuration declares its intended job, tools, side effects, output contract, and risk tier. That manifest selects the relevant evaluation packs.

  1. 01

    Platform core

    Isolation, state, retries, cancellation, human pause, approval gates, and telemetry. Required for every compatible configuration.

    UNIVERSAL
  2. 02

    Capability packs

    Research, writing, retrieval, planning, tools, orchestration, workflows, and external actions.

    MANIFEST-SELECTED
  3. 03

    Real user journeys

    Multi-step scenarios sampled from production patterns, including small human inputs mid-workflow.

    REALISTIC
  4. 04

    Incidents + private holdout

    Minimized failures, adversarial cases, and unpublished tasks that expose overfitting and regression.

    HARD TO GAME

GATE

Outcome success + safety

Did the real state change correctly? Were privacy, permission, and approval boundaries preserved?

Failure blocks release.

SCORE

Quality + reliability

Correctness, completeness, grounding, adherence, pass@1, passk, and paraphrase stability.

Report by capability and risk.

BUDGET

Cost + latency

Tokens, tool calls, sub-agents, dollars, time-to-first-event, completion p50/p95, and cost per success.

Fast failure is still failure.

05 / GRADING

Grade the outcome. Use the trace to explain it.

An agent saying “done” is not success. The evaluator checks the resulting state first, then inspects the trajectory for quality, safety, and efficiency.

INPUTTask + environment
RUNOutcome + trace
GRADERules + model + human
DECIDEShip, hold, investigate

Rule-based

Final state, schemas, exact counts, required fields, tool budgets, and approval invariants.

Model-based

Blinded task-specific rubrics for correctness, completeness, grounding, and plan quality.

Human / expert

Calibrates model graders, reviews high-impact cases, and settles disputes or ambiguous failures.

06 / RELEASE & REGRESSION

A change ships only when every layer stays healthy.

Default release gates

  1. 01

    Zero critical safety, privacy, or approval violations.

  2. 02

    Every critical outcome contract passes.

  3. 03

    Mean quality ≥75; no relevant capability pack below 70.

  4. 04

    95% success lower bound meets the risk-tier target.

  5. 05

    p95 latency and cost per success stay within budget.

Automatic regression alerts

−5 ppsuccess rate
−5 ptsquality or grounding
+20%p95 latency
+20%cost per success
+1new safety violation

The benchmark is also evaluated. Ambiguous prompts, brittle graders, contaminated tasks, and environment failures are quarantined—not silently counted as agent failures.
